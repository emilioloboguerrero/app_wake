2.1 — AUTHENTICATION FLOWS
================================================================================

PWA — AuthContext.js (apps/pwa/src/contexts/AuthContext.js)
─────────────────────────────────────────────────────────────
Flow:
  onAuthStateChanged fires → if user found, starts a check loop with delays:
  100ms, 300ms, 500ms, 1s, 2s (each check: if user is loaded, stop)
  Purpose: handle IndexedDB restore timing (Firebase auth restores from IndexedDB
  on web, which is async and not guaranteed to finish before the listener fires).

  Safety timeout: if user exists but loading still true after 2s, forces loading=false
  Safari fallback: if no user after 3s → loading=false
  Other browsers: if no user after 10s → loading=false

  Persistence: browserLocalPersistence (web), getReactNativePersistence(AsyncStorage) (native)
  Firebase Auth initialized differently per platform in apps/pwa/src/config/firebase.js

Exposes: { user, loading }
Auth methods in PWA: email/password, Google (googleAuthService), Apple (appleAuthService)

Creator Dashboard — AuthContext.jsx (apps/creator-dashboard/src/contexts/AuthContext.jsx)
──────────────────────────────────────────────────────────────────────────────────────────
Flow:
  onAuthStateChanged → getUser(firebaseUser.uid) → sets role, onboarding flags

Exposes: {
  user, userRole, loading,
  webOnboardingCompleted, profileCompleted, onboardingCompleted,
  isCreator, isAdmin,
  refreshUserData
}

Issues:
  - Has console.log on every render (performance concern, not using logger)
  - Blocks render with "Cargando..." while loading (no skeleton or partial load)

Native AppNavigator.js (apps/pwa/src/navigation/AppNavigator.js):
  After auth, fetches profile after 1500ms delay and caches to AsyncStorage.
  Uses firestoreService.getUser() for the profile fetch.

2.6 — OFFLINE / CACHING STRATEGY
================================================================================

Layer 1 — Full Course Download (low_ticket programs only):
  Service: courseDownloadService.js
  Storage: AsyncStorage
  Keys: course_data_{courseId}, course_download_meta_{courseId}, course_version_{courseId}
  Content: full module/session/exercise tree downloaded at assignment
  Version checking: compares version field, re-downloads on mismatch
  Background updates: checks for updates in background after serving cached data

Layer 2 — Hybrid Data Service (profile + courses metadata):
  Service: hybridDataService.js
  Storage: AsyncStorage, 24h TTL
  Keys: user_profile_v2, courses_data_v2, user_progress_v2, last_sync_timestamp_v2
  Pattern: return cache immediately → sync in background → update cache → re-render

Layer 3 — Workout Session Local-First:
  Service: workoutSessionService.js
  Storage: AsyncStorage (in-flight workout data)
  Keys: active_session, session_backup_0/1/2, session_metadata, upload_queue, pending_session_{id}
  Pattern: every set auto-saved to AsyncStorage; data uploaded after completion via upload queue

Layer 4 — In-Memory Caches (cleared on page reload):
  - consolidatedDataService: 5-min Map per userId
  - firestoreService._oneOnOneModulesCache: 5-min per courseId + in-flight deduplication
  - activityStreakService.streakCache: per userId+day
  - fatSecretTokenCache (Cloud Functions): per OAuth scope, refreshed 5min before expiry
  - WebAppNavigator profile: webStorageService (sessionStorage), 5min TTL

Layer 5 — React Query (Creator Dashboard + DailyWorkoutScreen):
  - staleTime: 30min for program data in DailyWorkoutScreen
  - queryClient.setQueryData() updated by onSnapshot in useProgramRealtime

Layer 6 — One-on-One Per-Session Fetch:
  No full download. Fetched on demand with 5-min in-memory cache (_oneOnOneModulesCache).
  Falls back to multiple Firestore paths in priority order:
    1. client_sessions (per-date session assignments)
    2. client_session_content (per-session content copies)
    3. client_plan_content (per-week content copies)
    4. plans/{planId}/modules/… (original plan)
    5. creator_libraries (resolved library references)

Firestore's own offline persistence:
  Not explicitly configured in any of the three apps.
  SDK default: offline persistence enabled on web when not in Node.js environment,
  but not explicitly set with enableIndexedDbPersistence() or enableMultiTabIndexedDbPersistence().

2.7 — PAYMENT & SUBSCRIPTION FLOW
================================================================================

MercadoPago Integration (Colombia, COP, server-side only via Cloud Functions)

One-time payment:
  1. Client calls createPaymentPreference (POST, no auth required beyond CORS)
     Body: { userId, courseId, returnUrl }
     Reads: courses/{courseId}, users/{userId}
     Returns: { init_point } (MP redirect URL)
     externalRef format: v1|{userId}|{courseId}|otp

Subscription:
  1. Client calls createSubscriptionCheckout (POST)
     Body: { userId, courseId, payer_email }
     Reads: courses/{courseId}, users/{userId}
     Writes: users/{userId}/subscriptions/{preapprovalId} (setDoc merge)
     Returns: { init_point, subscription_id }
     externalRef format: v1|{userId}|{courseId}|sub
     Rejects (409): wrong country (non-CO email domain) or wrong email

Webhook processing (processPaymentWebhook):
  Validates HMAC-SHA256 signature — two formats supported:
    Legacy: x-hmac-signature header (raw hex of body)
    New:    x-signature header with ts=…,v1=… (signs ts+body)

  Payment types handled:
    payment.created / payment.updated:
      - Fetches payment from MP API
      - Checks status: pending/in_process → 200 OK (wait); failed → mark processed; approved → assign
      - Uses Firestore transaction for atomic idempotency check on processed_payments/{paymentId}
      - Course assignment via atomic transaction on users/{userId}:
          courses[courseId] = { access_duration, expires_at, status: 'active', deliveryType, title, ... }
          purchased_courses: [...Set, courseId]

    subscription_authorized_payment.created:
      - Same as above but isSubscription=true
      - Extends expires_at for renewals (existing course → update expires_at)
      - Writes users/{userId}/subscriptions/{subscriptionId}

    subscription_preapproval:
      - Updates subscription status in users/{userId}/subscriptions/{preapprovalId}
      - Does NOT assign course (handled by authorized_payment event)

  Error handling:
    RETRYABLE errors → 500 (MP will retry)
    NON_RETRYABLE errors → 200 (write error to processed_payments)

Subscription management (updateSubscriptionStatus):
  Body: { userId, subscriptionId, action: 'cancel'|'pause'|'resume', survey? }
  Reads: users/{userId}/subscriptions/{subscriptionId}
  Calls: MP preapproval.update({ status: targetStatus })
  Writes: users/{userId}/subscriptions/{subscriptionId} (status, last_action, cancelled_at)
  If cancel + survey.answers present:
    Writes: subscription_cancellation_feedback/{id} (NEW COLLECTION — not in CLAUDE.md)
            Fields: userId, subscriptionId, answers, source, statusAfter, submittedAt,
                    courseId?, courseTitle?, statusBefore?, payerEmail?

All 9 functions in functions/src/index.ts:

1. createPaymentPreference
   Type: HTTPS onRequest
   Secrets: MERCADOPAGO_ACCESS_TOKEN
   Reads: courses/{courseId}, users/{userId}

   Type: HTTPS onRequest
   Secrets: MERCADOPAGO_ACCESS_TOKEN
   Reads: courses/{courseId}, users/{userId}
   Writes: users/{userId}/subscriptions/{preapprovalId}

3. processPaymentWebhook
   Type: HTTPS onRequest
   Secrets: MERCADOPAGO_WEBHOOK_SECRET, MERCADOPAGO_ACCESS_TOKEN
   Reads: courses/{courseId}, users/{userId}, processed_payments/{paymentId}
   Writes (transactional):
     - users/{userId} (courses map + purchased_courses)
     - users/{userId}/subscriptions/{subscriptionId}
     - processed_payments/{paymentId}
   HMAC validation: dual format (legacy + new x-signature)
   Retryable: 500 for network/API errors; 200 for validation/data errors

4. updateSubscriptionStatus
   Type: HTTPS onRequest
   Secrets: MERCADOPAGO_ACCESS_TOKEN
   Reads: users/{userId}/subscriptions/{subscriptionId}
   Writes: users/{userId}/subscriptions/{subscriptionId}
   Writes (if cancel): subscription_cancellation_feedback/{id}
   Calls: MP preapproval.update()

5. lookupUserForCreatorInvite
   Type: HTTPS onCall (requires auth)
   Reads: users/{creatorId} (role check), users by username query, users/{userId}
   Also: admin.auth().getUserByEmail() for email lookups
   Returns: { userId, displayName, email, username, age, gender, country, city, height, weight }
   Auth check: caller must have role 'creator' or 'admin'

7. nutritionFoodSearch
   Type: HTTPS onRequest
   Secrets: FATSECRET_CLIENT_ID, FATSECRET_CLIENT_SECRET
   External: FatSecret /rest/foods/search/v4 (OAuth scope: 'premier')
   Token: in-memory cache with 5min pre-expiry buffer
   No Firestore reads/writes

8. nutritionFoodGet
   Type: HTTPS onRequest
   Secrets: FATSECRET_CLIENT_ID, FATSECRET_CLIENT_SECRET
   External: FatSecret /rest/food/v5 (scope: 'basic' or 'premier' if include_sub_categories)
   No Firestore reads/writes

9. nutritionBarcodeLookup
   Type: HTTPS onRequest
   Secrets: FATSECRET_CLIENT_ID, FATSECRET_CLIENT_SECRET
   External: FatSecret /rest/food/barcode/find-by-id/v2 (scope: 'basic barcode')
   No Firestore reads/writes

10. sendEventConfirmationEmail
    Type: Firestore onCreate trigger
    Document: event_signups/{eventId}/registrations/{regId}
    Secrets: RESEND_API_KEY
    Reads: events/{eventId}
    Condition: event.settings.send_confirmation_email must be true (else skips)
    External: Resend API (HTML email with QR code)

================================================================================
2.9 — ERROR HANDLING PATTERNS
================================================================================

Cloud Functions:
  - classifyError helper: 'RETRYABLE' (network, API timeout) vs 'NON_RETRYABLE' (validation)
  - RETRYABLE → 500 status (MP/Pub-Sub will retry)
  - NON_RETRYABLE → 200 status (prevent retry loop)
  - All errors logged with functions.logger
  - Webhook errors: write processed_payments error doc to prevent future retries
  - Soft errors (survey write, checkout intent update) swallowed with try/catch + logger.error

PWA:
  - logger.js: debug/log/warn/error/prod levels, controlled by ?wake_debug=1 or localStorage.WAKE_DEBUG
  - No raw console.log in PWA services
  - firestoreService: fallback queries (orderBy → plain getDocs on index error)
  - exerciseHistoryService: getSessionHistoryPaginated has orderBy fallback
  - sessionRecoveryService: auto-complete <24h, discard >24h
  - authService (PWA): multiple delay checks + safety timeout for Safari/IndexedDB timing

Creator Dashboard:
  - Uses console.error/warn (no custom logger — inconsistency with PWA)
  - console.log present in many service methods (performance concern)
  - clientProgramService: 3-retry loop for addOneOnOneProgramToUserCourses with 500ms*attempt backoff
  - clientProgramService.assignProgramToClient: rollback on client_programs write failure
  - oneOnOneService: has try/catch fallback (query without orderBy if index missing)

================================================================================
2.12 — FIRESTORE SECURITY RULES AUDIT
================================================================================

Source: config/firebase/firestore.rules

Helper functions:
  isSignedIn()              — request.auth != null
  isOwner(userId)           — isSignedIn() && request.auth.uid == userId
  getUserRole()             — reads users/{auth.uid} via get() — costs 1 read per call!
  isAdmin()                 — getUserRole() == 'admin'
  isCreator()               — getUserRole() == 'creator' OR isAdmin()
  creatorHasClientAccess(u) — isCreator() && exists(creator_client_access/{uid}_{u})

NOTE: getUserRole() does a Firestore document read on EVERY rule evaluation that calls
isCreator() or isAdmin(). This means multiple reads per request in complex rules.

EXPLICIT RULES (what they cover):
  users/{userId}                      — owner/admin/creatorHasClientAccess read+update
  users/{userId}/exerciseHistory      — owner only
  users/{userId}/sessionHistory       — owner write; owner/admin/creator read

[CRITICAL] catch-all gives any logged-in user read/write to:
  - client_programs: user A can read user B's personalized program assignments
  - client_sessions: any user can read all clients' session schedules
  - subscription_cancellation_feedback: any user can read all cancellation survey answers

[HIGH] exercises_library Storage write is open to ALL authenticated users

[HIGH] client_nutrition_plan_content rule uses get() on nutrition_assignments for each check

[MEDIUM] getUserRole() reads users/{uid} on EVERY rule invocation

2.16 — DATA DELIVERY PATTERNS
================================================================================

Low-ticket program delivery:
  1. User purchases → Cloud Function writes users/{userId}.courses[courseId]
  2. PWA courseDownloadService downloads full course tree to AsyncStorage
  3. hybridDataService serves AsyncStorage cache (24h TTL) + background Firestore sync
  4. consolidatedDataService wraps: purchase check + hybrid + download data in 5-min cache
  5. DailyWorkoutScreen serves pre-cached session from AsyncStorage (offline capable)

One-on-one program delivery:
  1. Creator assigns program via clientProgramService.assignProgramToClient
     → writes client_programs + users.courses (deliveryType: 'one_on_one')
  2. Creator assigns plan to weeks via assignPlanToWeek
     → writes planAssignments map on client_programs doc
     → writes client_sessions docs (per-session per-date)
  3. PWA firestoreService.getCourseModules reads:
     a. client_sessions → find session for today's date
     b. client_session_content → get personalized session copy (if exists)
     c. client_plan_content → get personalized week copy (if exists)
     d. plans/{planId}/modules → original plan content
     e. creator_libraries → resolve any library references
  4. In-memory 5-min cache prevents repeat reads within same session

Weekly program delivery:
  Similar to low-ticket but modules have week field (ISO week strings like "2025-W03")
  courseDownloadService filters modules by current week

Nutrition delivery:
  1. getEffectivePlanForUser: checks nutrition_assignments (by userId, sorted startDate desc)
  2. Finds active assignment for requested date
  3. Resolution order: client_nutrition_plan_content → assignment.plan snapshot → creator library plan
  4. Diary: direct read/write to users/{userId}/diary with date filter

Session completion flow:
  1. workoutSessionService saves all set data to AsyncStorage during workout
  2. On complete: marks session in AsyncStorage + adds to upload_queue
  3. uploadService processes queue: calls exerciseHistoryService.addSessionData()
  4. exerciseHistoryService writes:
     - users/{userId}/sessionHistory/{sessionId}
     - users/{userId}/exerciseHistory/{exerciseKey} (prepend)
     - users/{userId}/exerciseLastPerformance/{exerciseKey} (if newer)
  5. sessionRecoveryService handles crashes: auto-completes sessions <24h old

Content copy pattern (one-on-one personalization):
  Creator can create copies of content for specific clients:
  - client_plan_content/{userId}_{programId}_{weekKey}: week-level personalization
  - client_session_content/{id}: session-level personalization
  - client_nutrition_plan_content/{assignmentId}: nutrition plan copy
  PWA always checks for copy first, falls back to original

  2. client_programs: any auth user can read all clients' personalized program data

ARCHITECTURAL OBSERVATIONS:
─────────────────────────────
  - The collection client_programs uses {userId}_{programId} as doc ID — this prevents
    efficient querying "all programs for a user" without knowing program IDs in advance.
    The creator needs to know program IDs to look up client programs.

  - The events collection has two update rules (one for creator/admin, one for registration_count)
    Firestore evaluates rules with OR — both are valid simultaneously. This is intentional and
    correctly implemented.

  - Library resolution (libraryModuleRef/librarySessionRef) adds async waterfall reads
    on every course load for one-on-one programs. The 5-min in-memory cache helps but
    cold loads can generate many Firestore reads.

  - No explicit Firestore offline persistence configured. Firestore SDK default behavior
    enables indexedDB persistence on web automatically, but this is not confirmed tested.

  - The FatSecret token cache is in Cloud Function memory (process-level). In a scaled
    multi-instance scenario, each cold start re-fetches a token. This is correct behavior
    (stateless function) but means potentially many token requests if functions scale widely.

  - MercadoPago external_reference format v1|{userId}|{courseId}|{type} — the v1 prefix
    suggests future versioning was anticipated. Good for forward compatibility.

  - sendEventConfirmationEmail has no idempotency guard — if a registration document fires
    onCreate twice (unlikely but possible in Firestore trigger scenarios), two emails go out.
    Consider writing a sent flag to the registration doc after successful send.

  Files: apps/creator-dashboard/src/screens/EventEditorScreen.jsx
